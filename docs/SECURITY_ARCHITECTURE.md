# Legacy Vault - 安全架构设计文档

> 本文档定义了 Legacy Vault 的完整安全架构。安全是本产品的第一优先级，所有设计决策必须以安全为前提。

---

## 1. 安全设计原则

| 原则 | 说明 |
|------|------|
| 零知识 | 服务端永远不接触明文数据和密钥，无法解密任何用户内容 |
| 端到端加密 | 数据在用户设备上加密，仅在用户设备上解密 |
| 最小权限 | 每个组件只拥有完成其功能所需的最小权限 |
| 纵深防御 | 多层安全机制，单点失败不导致系统性安全崩溃 |
| 密码学敏捷性 | 架构支持算法升级，不绑定单一加密算法 |
| 可审计性 | 所有安全相关操作留有审计日志（加密存储） |
| 前向保密 | 历史通信数据即使密钥泄露也不可追溯解密 |

## 2. 加密体系总览

### 2.1 密钥层级架构

```
用户主密码 (User Master Password)
    |
    | Argon2id KDF
    | (m=47104/46MiB, t=1, p=1, salt=16字节随机)
    v
主密钥 (Master Key, 256-bit AES key)
    |
    +---> 加密密钥 (Encryption Key, 256-bit)
    |       用于加密/解密 数据加密密钥(DEK)
    |
    +---> 认证密钥 (Authentication Key, 256-bit)
    |       HMAC-SHA256，用于向服务端证明身份（不暴露主密码）
    |
    +---> 继承托管密钥 (Inheritance Escrow Key, 256-bit)
            用于生成 Shamir 分片

            主密钥
              |
         +----+----+
         v         v
    加密密钥   认证密钥
         |
    +----+----+----+---- ... ----+
    v    v    v    v              v
  DEK1 DEK2 DEK3 DEK4  ...    DEKn    (每条数据独立)
    |    |    |    |              |
    v    v    v    v              v
  数据1 数据2 数据3 数据4  ...  数据n   (AES-256-GCM加密)
```

### 2.2 密钥派生详细流程

**步骤1：从主密码派生主密钥**
```
输入：用户主密码 (UTF-8编码)
盐值：16字节密码学安全随机数 (crypto_secretbox_NONCEBYTES)
算法：Argon2id
参数：
  - 内存成本 (m)：47104 KiB (46 MiB)
  - 时间成本 (t)：1 次迭代
  - 并行度 (p)：1
  - 输出长度：32 字节 (256 bit)
输出：主密钥 (Master Key)

注：以上参数遵循 OWASP 2025 推荐。随硬件发展需定期评估和升级。
备选参数：m=19456 (19 MiB), t=2, p=1（等效安全级别，CPU更高/内存更低）
```

**步骤2：从主密钥派生子密钥**
```
加密密钥 = HKDF-SHA256(主密钥, info="enc", salt=随机盐)
认证密钥 = HKDF-SHA256(主密钥, info="auth", salt=随机盐)
继承托管密钥 = HKDF-SHA256(主密钥, info="inherit", salt=随机盐)
```

**步骤3：数据加密**
```
对于每条数据：
  1. 生成随机DEK (256-bit)
  2. 生成随机Nonce (96-bit / 12字节，GCM标准)
  3. 密文 = AES-256-GCM.Encrypt(DEK, Nonce, 明文, AAD=元数据哈希)
  4. 加密后的DEK = AES-256-GCM.Encrypt(加密密钥, Nonce2, DEK)
  5. 存储：{加密后的DEK, Nonce, Nonce2, 密文, AAD}
```

### 2.3 加密算法选型

| 用途 | 算法 | 标准/来源 | 说明 |
|------|------|-----------|------|
| 密钥派生 | Argon2id | RFC 9106, PHC优胜 | 抗GPU和侧信道攻击 |
| 对称加密 | AES-256-GCM | NIST SP 800-38D | 认证加密(AEAD)，硬件加速 |
| 非对称加密 | X25519 | RFC 7748 | 用于继承人密钥交换 |
| 数字签名 | Ed25519 | RFC 8032 | 用于审计日志签名 |
| 哈希 | SHA-256 / BLAKE2b | FIPS 180-4 / RFC 7693 | 数据完整性校验 |
| 密钥协商 | HKDF-SHA256 | RFC 5869 | 从主密钥派生子密钥 |
| 秘密共享 | Shamir SSS (GF(2^256)) | Adi Shamir, 1979 | 继承人密钥分片 |
| 随机数生成 | CSPRNG | OS原生 | window.crypto.getRandomValues() |

## 3. Shamir 秘密共享方案

### 3.1 概述

Shamir秘密共享(SSS)用于将用户的继承托管密钥分割为n份分片(shares)，其中任意t份分片可重建原始密钥，而少于t份分片在信息论意义上不泄露任何关于密钥的信息（完美保密性）。

### 3.2 参数设计

| 参数 | 值 | 说明 |
|------|-----|------|
| 有限域 | GF(p), p = 2^256 - 189 (256-bit素数) | 大于任何可能的密钥值 |
| 秘密长度 | 256 bit | 继承托管密钥长度 |
| 多项式次数 | t - 1 | t为阈值 |
| 系数生成 | CSPRNG | 密码学安全随机数 |
| 分片编码 | (x, y) 对，x为分片索引，y为256-bit值 | 每个分片约64字节 |

### 3.3 推荐阈值配置

| 场景 | n (总分片数) | t (阈值) | 说明 |
|------|-------------|----------|------|
| 个人基础 | 3 | 2 | 配偶+子女+律师，任意2人可重建 |
| 家庭标准 | 5 | 3 | 多个家庭成员+律师，任意3人可重建 |
| 高安全 | 7 | 4 | 多信任方分散持有，任意4人可重建 |
| 企业级 | 10 | 6 | 企业关键人物分散持有 |

### 3.4 分片生命周期

```
1. 生成阶段
   用户设置继承计划 → 生成继承托管密钥
   → 选择(t, n)参数 → 生成随机多项式 f(x)
   → 计算n个分片: share_i = (i, f(i))
   → 每个分片用对应继承人的公钥加密

2. 分发阶段
   → 每个继承人收到自己的加密分片
   → 继承人用自己的私钥解密并安全存储分片
   → 系统记录分片分发状态（不记录分片内容）

3. 重建阶段（触发条件满足后）
   → 继承人提交各自的分片
   → 系统收集到 >= t 个有效分片
   → 使用拉格朗日插值重建继承托管密钥
   → 用继承托管密钥解密数据加密密钥
   → 用数据加密密钥解密对应数据
   → 重建在客户端完成，服务端不接触明文分片

4. 轮换阶段
   → 用户修改继承计划时重新生成多项式
   → 旧分片自动失效
   → 分发新分片给继承人
```

### 3.5 分片安全增强

- **分片加密存储**：每个分片在继承人设备上用继承人自己的密钥加密存储
- **分片完整性校验**：每个分片附带HMAC签名，防篡改
- **分片时效性**：分片包含时间戳，支持过期机制
- **分片版本控制**：每次轮换递增版本号，旧版本分片自动拒绝
- **反串谋保护**：即使t-1个继承人串谋，也无法获得任何关于密钥的信息

## 4. 零知识架构详细设计

### 4.1 服务端存储的内容

| 数据项 | 服务端存储形式 | 服务端能否解密 |
|--------|---------------|---------------|
| 用户主密码 | 不存储 | 不适用 |
| 主密钥 | 不存储 | 不适用 |
| 加密密钥/认证密钥 | 不存储 | 不适用 |
| 盐值 (salt) | 明文存储 | 不适用（不含敏感信息） |
| 认证哈希 | 存储（用于验证身份） | 仅验证，无法反推密码 |
| 数据加密密钥(DEK) | 加密存储 | 否 |
| 用户数据 | 加密存储 | 否 |
| 元数据 | 最小化存储（类型、大小、时间戳） | 仅结构化元数据 |
| Shamir分片 | 不经过服务端 | 不适用 |
| 审计日志 | 加密存储 | 否 |

### 4.2 认证流程（零知识）

```
客户端：
  1. auth_key = HKDF-SHA256(master_key, info="auth")
  2. auth_hash = SHA256(auth_key) → 发送给服务端注册

登录时：
  1. 客户端从服务端获取盐值
  2. 客户端计算 master_key = Argon2id(password, salt)
  3. 客户端计算 auth_key = HKDF-SHA256(master_key, info="auth")
  4. 客户端计算 auth_hash = SHA256(auth_key)
  5. 客户端将 auth_hash 发送给服务端
  6. 服务端比较 auth_hash 是否匹配

注：服务端只接收 auth_hash，无法反推 auth_key 或 master_key 或 password
```

### 4.3 数据同步的零知识保证

```
上传：
  客户端加密数据 → 上传密文 → 服务端存储密文
  服务端看到的只是不可解读的字节流

下载：
  服务端返回密文 → 客户端解密 → 用户看到明文
  解密全程在客户端完成

搜索：
  客户端下载所有加密元数据 → 客户端本地解密 → 客户端本地搜索
  服务端不参与搜索过程
```

## 5. 威胁模型分析 (Threat Model)

### 5.1 资产清单

| 资产 | 敏感级别 | 说明 |
|------|---------|------|
| 用户密码库 | 极高 | 所有存储的密码 |
| 私密文件 | 极高 | 遗嘱、法律文件、私人信件 |
| 加密货币私钥/助记词 | 极高 | 直接关联财产损失 |
| 个人媒体 | 高 | 私密照片、视频、录音 |
| 继承人信息 | 高 | 继承人身份和联系方式 |
| Shamir分片 | 极高 | 可能用于重建主密钥 |
| 用户主密码 | 极高 | 一切安全的根基 |
| 审计日志 | 中 | 操作历史记录 |

### 5.2 威胁行为者

| 行为者 | 能力 | 动机 |
|--------|------|------|
| 外部攻击者 | 网络攻击、社会工程 | 窃取数据、勒索 |
| 内部人员(我方) | 访问服务端基础设施 | 窥探用户数据(零知识应阻止) |
| 国家级行为者 | 高级持续威胁、法律强制 | 监控、执法 |
| 恶意继承人 | 社会工程、物理接触 | 提前获取遗产 |
| 竞争对手 | 商业情报 | 窃取技术/用户数据 |

### 5.3 攻击向量与防御措施

#### 攻击向量1：服务端数据泄露
- **攻击场景**：攻击者入侵服务器，获取数据库
- **影响**：获取所有加密后的数据
- **防御**：
  - 零知识架构：即使完整数据库泄露，攻击者只获得密文
  - AES-256-GCM加密：暴力破解不可行（2^256种可能）
  - 每条数据独立DEK：一个DEK泄露不影响其他数据
  - 数据库自身加密（静态加密）
- **残留风险**：低。元数据（条目数量、大小、时间戳）可能泄露使用模式

#### 攻击向量2：主密码暴力破解
- **攻击场景**：攻击者获取盐值和auth_hash，尝试暴力破解
- **影响**：获取主密钥，解密所有数据
- **防御**：
  - Argon2id：内存硬函数，抗GPU/ASIC加速
  - 高成本参数（46MiB内存）：单次尝试成本高
  - 服务端限速：登录失败5次后启动验证码，20次后锁定账号
  - 强密码策略：注册时强制12位以上，含大小写+数字+符号
- **残留风险**：中。极弱密码仍有风险（依赖用户选择强密码）

#### 攻击向量3：客户端恶意软件
- **攻击场景**：用户设备被植入键盘记录器或内存读取恶意软件
- **影响**：主密码和解密后的明文被窃取
- **防御**：
  - 生物识别认证（减少密码输入频率）
  - WebAuthn/FIDO2硬件密钥支持
  - 内存中敏感数据及时清零
  - 安全输入框（防截屏/防剪贴板监控）
  - 设备信任管理（异常设备登录警告）
- **残留风险**：高。客户端安全最终取决于设备环境，超出应用控制范围

#### 攻击向量4：中间人攻击 (MITM)
- **攻击场景**：攻击者拦截客户端与服务端的通信
- **影响**：窃取认证令牌，篡改数据
- **防御**：
  - 强制TLS 1.3（HSTS + Certificate Pinning）
  - 端到端加密：即使TLS被破解，数据仍是密文
  - 证书透明度(CT)监控
- **残留风险**：低

#### 攻击向量5：恶意继承人攻击
- **攻击场景**：继承人试图在用户存活时获取数据
- **影响**：提前获得遗产内容
- **防御**：
  - Shamir阈值：单个继承人无法重建密钥
  - 生存验证：用户存活时任何继承人无法触发继承
  - 紧急访问等待期：用户有充足时间拒绝不当请求
  - 审计日志：所有继承人操作可追溯
- **残留风险**：低-中。如果多个继承人串谋且达到阈值，理论上可发起虚假继承声明（通过多人确认机制缓解）

#### 攻击向量6：供应链攻击
- **攻击场景**：攻击者篡改依赖库，注入恶意代码
- **影响**：加密过程被破坏，密钥被窃取
- **防御**：
  - 依赖锁定（lock files）和完整性校验
  - 最小化依赖，核心加密代码自研/审计
  - 定期安全依赖扫描
  - 可重现构建(Reproducible Builds)
  - 代码签名和发布验证
- **残留风险**：中

#### 攻击向量7：社会工程攻击
- **攻击场景**：攻击者冒充客服/继承人/律师获取信息
- **影响**：用户泄露主密码或恢复密钥
- **防御**：
  - 零知识架构：客服无法"帮助重置密码"，因为我们也没有密钥
  - 用户教育：明确告知"我们永远不会索要你的密码"
  - 继承人身份多重验证
- **残留风险**：中。社会工程的防御最终依赖用户安全意识

#### 攻击向量8：量子计算威胁
- **攻击场景**：未来量子计算机破解现有加密算法
- **影响**：AES-256在量子场景下等效安全性降至128-bit（仍安全），但非对称加密(X25519/Ed25519)可能被Shor算法破解
- **防御**：
  - 密码学敏捷性：架构支持算法升级
  - 后量子加密算法储备：关注NIST PQC标准化成果（ML-KEM/ML-DSA/SLH-DSA）
  - AES-256本身在量子场景下仍足够安全（Grover算法仅将暴力搜索加速至2^128）
  - 路线图包含向后量子密码学迁移的计划
- **残留风险**：低（短期），中（长期10-15年）

## 6. 密钥管理生命周期

### 6.1 密钥创建

```
注册流程：
  1. 用户输入主密码
  2. 生成16字节随机盐值
  3. Argon2id(密码, 盐值) → 主密钥
  4. HKDF派生加密密钥、认证密钥、继承托管密钥
  5. 主密码和主密钥仅存在于内存中
  6. 盐值和auth_hash上传至服务端
  7. 加密密钥和继承托管密钥的加密包裹上传至服务端（用主密钥加密）
```

### 6.2 密钥使用

```
日常使用：
  1. 用户输入主密码（或生物识别触发密钥链中的缓存密钥）
  2. 派生主密钥和加密密钥
  3. 解密需要的DEK
  4. 用DEK解密具体数据
  5. 操作完成后清理内存中的敏感密钥

关键安全规则：
  - 主密钥在内存中的停留时间最小化
  - 非活动状态5分钟后自动锁定（用户可配置1-30分钟）
  - 应用切换到后台时清理内存中的明文数据
  - 每个会话使用独立的会话密钥
```

### 6.3 密钥轮换

```
主密码修改：
  1. 验证当前主密码
  2. 用新密码生成新的主密钥（新盐值）
  3. 用新主密钥重新加密加密密钥的包裹
  4. 更新auth_hash
  5. DEK不需要变更（仅加密密钥的包裹变更）
  6. 如果设置了继承计划，重新生成Shamir分片并分发

定期轮换建议：
  - 建议用户每12个月修改主密码
  - 继承人变更时强制重新分片
  - DEK不需要定期轮换（每条数据独立，风险隔离）
```

### 6.4 密钥销毁

```
账号删除：
  1. 用户确认删除（需主密码 + 邮件确认）
  2. 服务端删除所有加密数据和DEK
  3. 通知所有继承人分片已失效
  4. 服务端删除盐值和auth_hash
  5. 30天内可恢复（之后永久删除）

数据安全擦除：
  - 内存：敏感数据使用sodium_memzero()安全清零
  - 存储：使用安全删除，覆写后删除
  - 服务端：加密数据删除后密钥销毁，即使备份中残留密文也不可解密
```

### 6.5 密钥恢复

```
恢复密钥（Recovery Kit）：
  1. 注册时生成24个单词的助记词（BIP39标准）
  2. 助记词可派生出恢复密钥
  3. 恢复密钥用于加密主密钥的一个备份副本
  4. 备份副本存储在服务端

恢复流程：
  1. 用户选择"忘记密码"
  2. 邮箱验证身份
  3. 输入24个助记词
  4. 系统用恢复密钥解密主密钥备份
  5. 用户设置新主密码
  6. 用新主密码重新加密主密钥
  7. 重新生成恢复密钥（旧助记词失效）

注意：如果用户同时丢失主密码和恢复密钥，数据将永久无法恢复。
这是零知识架构的代价，也是安全保证的基础。
```

## 7. 传输安全

### 7.1 网络层安全

| 措施 | 详细 |
|------|------|
| TLS版本 | 强制TLS 1.3 |

## 10. 后量子密码学迁移路线

### 10.1 当前状态评估

| 组件 | 当前算法 | 量子威胁 | 迁移优先级 |
|------|---------|---------|-----------|
| 对称加密 | AES-256-GCM | 低（等效128-bit安全） | 低 |
| 密钥派生 | Argon2id | 低 | 低 |
| 密钥交换 | X25519 | 高（Shor算法） | 高 |
| 数字签名 | Ed25519 | 高（Shor算法） | 高 |
| 哈希函数 | SHA-256/BLAKE2b | 中（Grover加速） | 低 |

### 10.2 迁移计划

- **短期（2026-2027）**：关注NIST PQC标准化成果，评估ML-KEM (CRYSTALS-Kyber)和ML-DSA (CRYSTALS-Dilithium)
- **中期（2027-2028）**：实现混合模式（经典算法 + 后量子算法并行），确保向后兼容
- **长期（2028+）**：完全迁移至后量子算法（如果量子计算威胁变为现实）
