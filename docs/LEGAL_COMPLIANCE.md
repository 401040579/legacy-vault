# Legacy Vault - 法律合规文档

> 遗产传承产品涉及广泛的法律问题。本文档梳理各主要市场的法律环境、合规要求和风险点，指导产品设计和运营决策。

---

## 1. 数字遗产法律概览

### 1.1 什么是"数字遗产"

数字遗产包括但不限于：
- **数字账号**：社交媒体、邮箱、云存储、订阅服务
- **数字资产**：加密货币、数字艺术(NFT)、域名、网店
- **数字内容**：照片、视频、文档、博客、电子书
- **数字身份**：密码、密钥、身份验证凭证
- **虚拟财产**：游戏账号、虚拟物品、积分

### 1.2 核心法律问题

| 问题 | 说明 | 对产品的影响 |
|------|------|-------------|
| 数字遗产是否属于"遗产"？ | 各国法律定义不同 | 决定继承功能的法律基础 |
| 电子遗嘱是否有法律效力？ | 大多数国家要求书面签名 | 影响"法律文档模板"功能定位 |
| 隐私权与继承权的冲突 | 死者通信秘密 vs 继承人知情权 | 影响内容分发的合规设计 |
| 服务协议中的反继承条款 | 许多平台禁止账号转让 | 我们仅存储/传递凭证，不直接操作第三方账号 |
| 跨境数据传输 | 用户和继承人可能在不同国家 | 影响数据存储和传输架构 |
| 数据保留期限 | 用户去世后数据保存多久？ | 需要明确的保留和删除策略 |

## 2. 各国/地区法律环境分析

### 2.1 中国

**现行法律框架：**
- **《民法典》第127条**（2021年生效）：明确"法律对数据、网络虚拟财产的保护有规定的，依照其规定"。这是数字遗产保护的法律基石，但具体细则尚待完善。
- **《民法典》第1122条**：遗产是自然人死亡时遗留的个人合法财产。数字财产如属于"合法财产"，可被继承。
- **《个人信息保护法》（PIPL）**：中国版GDPR，对个人信息的收集、存储、传输有严格要求。

**数字遗产继承现状：**
- 2024年中华遗嘱库接收458份涉及虚拟财产的遗嘱，同比增长217%
- 微信账号遗嘱占比达41%，超越传统房产（28%）
- 司法实践中数字遗产纠纷不断增加，但缺乏明确的程序规范
- 网络服务协议中常见的"账号不可继承"条款的法律效力存在争议

**对产品的影响：**
- 服务中国用户需遵守PIPL，数据存储于中国境内或通过安全评估
- 产品可定位为"私密信息传承工具"而非"法律遗嘱工具"，避免法律效力争议
- 需在用户协议中明确：本产品存储和传递的是用户的私人信息，不构成法律遗嘱

### 2.2 美国

**现行法律框架：**
- **RUFADAA (统一受信人数字资产访问法)**：已被约50个州采纳。允许受信人（遗嘱执行人、信托受托人）在一定条件下访问死者的数字资产。用户可以通过"在线工具"（如我们的产品）指定谁可以访问其数字资产。
- **CCPA/CPRA (加州消费者隐私法)**：加州消费者隐私权利，包括数据删除权。
- **各州遗嘱法**：电子遗嘱的效力因州而异。少数州（如内华达、印第安纳、佛罗里达等）承认电子遗嘱。

**对产品的影响：**
- RUFADAA为我们的产品提供了法律基础——用户可以通过我们的"在线工具"指定数字资产的访问权限
- 需遵守CCPA，提供数据删除选项
- "法律文档模板"功能需要按州定制，并附免责声明

### 2.3 欧盟

**现行法律框架：**
- **GDPR 第27条附注**：GDPR不适用于已故人员的个人数据。但成员国可自行制定关于死者数据处理的规则。
- **法国**：已在国家数据保护法中增加了死者数字遗产条款，允许继承人行使对死者数据的某些权利。
- **意大利**：允许有合法理由的人访问死者的数字数据。
- **西班牙**：《有机数字法》允许继承人访问死者的数字内容（除非死者生前明确禁止）。

**对产品的影响：**
- GDPR合规是进入欧盟市场的前提（即使不直接适用于死者数据）
- 需要数据处理协议(DPA)和隐私影响评估(DPIA)
- 数据存储需考虑欧盟数据本地化要求（欧盟境内或充分性决定国家）
- 用户需有明确的"数据主体权利"行使途径

### 2.4 日本

**现行法律框架：**
- 日本目前没有专门的数字遗产法规
- 《个人信息保护法（APPI）》不适用于已故人员的个人数据
- 数字资产继承主要依赖传统继承法和各平台的服务条款

**对产品的影响：**
- 法律环境相对宽松，但缺乏明确保障
- 日本是高龄化社会，数字遗产需求强烈，市场机会大
- 建议与日本律师事务所合作，提供本地化法律指导

### 2.5 韩国

- 《个人信息保护法（PIPA）》对数据保护要求严格
- 数字遗产继承无专门法规，依赖传统继承法
- 高度数字化社会，市场潜力大

### 2.6 法律环境对比总结

| 维度 | 中国 | 美国 | 欧盟 | 日本 |
|------|------|------|------|------|
| 数字遗产立法 | 框架性（民法典127条） | 较完善（RUFADAA） | 碎片化（各成员国） | 缺失 |
| 电子遗嘱效力 | 不被承认 | 部分州承认 | 多数不承认 | 不被承认 |
| 数据保护法 | PIPL | CCPA/CPRA | GDPR | APPI |
| 数据本地化 | 严格 | 宽松 | 中等 | 宽松 |
| 跨境传输 | 需安全评估 | 相对自由 | 需充分性决定/SCCs | 需适当保障 |

## 3. 电子遗嘱的法律效力

### 3.1 现状

**大多数国家/地区不承认纯电子遗嘱的法律效力。** 传统遗嘱通常要求：
- 遗嘱人亲笔签名（自书遗嘱）
- 至少两名见证人在场（代书遗嘱/公证遗嘱）
- 书面形式（纸质）

### 3.2 产品定位策略

Legacy Vault **不应将自己定位为"电子遗嘱工具"**，而应定位为：

> **"安全的私人信息传承工具"** —— 帮助用户安全存储和传递私密信息，作为正式遗嘱的补充而非替代。

具体策略：
1. **法律文档模板**定位为"参考模板"，用户使用后仍需经过正式的法律程序（公证/律师见证）
2. **用户协议**中明确声明：本产品提供的内容传递服务不构成法律意义上的遗嘱
3. **合作律师服务**：提供与律师事务所对接的增值服务，帮助用户将Legacy Vault中的规划转化为合法遗嘱
4. **公证对接**：v2.0版本考虑与在线公证服务对接

### 3.3 免责声明模板

在产品中显著位置展示：
> "Legacy Vault 帮助您安全存储和传递私密信息。本产品存储的内容和设定的传递规则不构成法律意义上的遗嘱或信托文件。如需制定具有法律约束力的遗嘱，请咨询专业律师。"

## 4. 隐私权与继承权的平衡

### 4.1 冲突场景

| 场景 | 冲突点 | 产品应对 |
|------|--------|---------|
| 死者的私密通信 | 通信秘密权 vs 继承人知情权 | 尊重用户生前的分配设定 |
| 数字账号密码 | 隐私权 vs 财产继承权 | 仅传递用户明确授权的内容 |
| 加密货币私钥 | 资产继承 vs 安全风险 | 通过Shamir分片安全传递 |
| 社交媒体账号 | 人格权 vs 纪念需求 | 仅传递密码，不直接操作账号 |

### 4.2 设计原则

1. **用户主权原则**：用户完全控制什么内容给什么人。系统只执行用户的意愿，不做任何"默认"分配。
2. **最小传递原则**：只传递用户明确授权的内容，不传递未分配的内容。
3. **不可推定原则**：如果用户未设置遗产计划就去世，系统不应假设任何人有权访问数据。未分配的数据按保留策略处理（见第5节）。
4. **透明原则**：用户可以随时查看和修改继承计划。继承人可以看到"你被分配了某些内容"，但无法在触发前看到具体内容。

## 5. 数据保留与删除策略

### 5.1 活跃用户数据

| 数据类型 | 保留策略 |
|----------|---------|
| 加密数据（保险箱内容） | 用户不删除则永久保留 |
| 账号信息 | 账号存续期间 |
| 审计日志 | 滚动保留2年 |
| 访问日志 | 滚动保留90天 |
| 客户支持通信 | 解决后保留1年 |

### 5.2 已故用户数据

| 场景 | 保留策略 |
|------|---------|
| 已触发继承且所有内容已被接收 | 继承人确认接收后保留90天，之后安全删除 |
| 已触发继承但部分内容未被接收 | 保留5年，每年通知继承人 |
| 未设置继承计划 | 按"不活跃账号"策略处理 |
| 继承人主动要求删除 | 继承人接收内容后可请求删除源数据 |

### 5.3 不活跃账号策略

```
连续12个月无登录且未设置遗产计划
→ 发送"账号不活跃"提醒（邮件+短信）
→ 30天无响应 → 发送第二次提醒
→ 再30天无响应 → 账号标记为"休眠"
→ 休眠12个月后 → 发送"即将删除"最终通知
→ 30天无响应 → 安全删除所有数据

注：已设置遗产计划的账号不受此策略影响，
因为"不活跃"可能恰恰意味着遗产计划需要被触发。
```

### 5.4 用户数据删除权

根据GDPR/CCPA/PIPL等法规，用户有权要求删除个人数据：
- **删除请求流程**：主密码验证 → 确认删除范围 → 通知继承人分片将失效 → 执行删除 → 30天内可恢复 → 永久删除
- **不可删除的例外**：法律要求保留的数据（如财务交易记录）、正在执行中的继承流程涉及的数据

## 6. 跨境合规

### 6.1 数据存储架构

考虑各国数据本地化要求，建议采用分区存储：

| 地区 | 数据中心位置 | 合规要求 |
|------|-------------|---------|
| 中国大陆用户 | 中国大陆 | PIPL + 数据本地化 |
| 欧盟用户 | 欧盟（如德国/爱尔兰） | GDPR |
| 美国用户 | 美国 | CCPA |
| 亚太其他 | 新加坡/日本 | 各国法规 |

### 6.2 跨境数据传输

当用户和继承人不在同一法域时：
- 继承人接收的是**加密数据**，解密在继承人本地完成
- 零知识架构意味着跨境传输的是密文，降低了数据泄露的合规风险
- 但仍需确保传输机制符合数据出境要求（中国需安全评估/标准合同，欧盟需SCCs/充分性决定）

### 6.3 本地化合规清单

- [ ] 各目标市场的隐私政策本地化翻译
- [ ] 各目标市场的用户协议法律审查
- [ ] 数据处理协议(DPA)准备
- [ ] 数据保护影响评估(DPIA)完成
- [ ] 各目标市场的Cookie政策
- [ ] 数据出境安全评估（中国市场）
- [ ] 标准合同条款(SCCs)签署（欧盟市场）
- [ ] 数据保护官(DPO)任命（欧盟市场必需）

## 7. 知识产权与内容合规

### 7.1 用户上传内容

- 用户上传的内容由用户自行负责其合法性
- 零知识加密意味着我们无法审查内容（既是技术优势也是合规挑战）
- 用户协议中需明确：不得上传违法内容
- 如收到合法的执法请求，我们技术上无法提供内容明文（这是功能，不是缺陷）

### 7.2 执法请求应对

```
收到合法的数据披露请求（法院命令/搜查令）
→ 法律团队审查请求合法性
→ 通知用户（如法律允许）
→ 提供我们拥有的数据：
    - 账号元数据（注册邮箱、创建时间、最后登录时间）
    - 加密后的数据（密文）
    - 无法提供：明文数据、主密码、解密密钥
→ 发布透明度报告（年度汇总）
```

### 7.3 透明度报告

每年发布透明度报告，披露：
- 收到的执法请求数量
- 提供数据的请求数量
- 按国家/地区分类
- 请求类型（用户数据/内容删除/其他）

## 8. 服务持续性保障

### 8.1 公司运营中止时的用户权益保障

这是用户最关心的法律问题之一：**如果Legacy Vault公司倒闭，我的数据怎么办？**

**法律层面保障：**
- 用户协议中写入"数据可携带权"条款
- 承诺至少提前6个月通知用户服务终止
- 提供数据导出工具（加密或解密格式）
- 考虑设立"数据托管信托"（详见 RISKS.md）

**技术层面保障：**
- 客户端可独立工作（PWA/本地存储模式）
- 开源核心加密代码，用户可自行解密
- 标准化数据格式（JSON + 标准加密算法）

### 8.2 用户协议关键条款

| 条款 | 内容 | 合规依据 |
|------|------|---------|
| 数据所有权 | 用户数据归用户所有，我们仅提供存储和传递服务 | GDPR Art.20 |
| 服务终止通知 | 至少6个月提前通知 | 行业最佳实践 |
| 数据可携带 | 用户可随时导出全部数据 | GDPR Art.20 |
| 免责声明 | 非法律遗嘱替代品 | 法律风险控制 |
| 零知识声明 | 明确告知我们无法帮助恢复密码 | 用户知情权 |
| 继承人权利 | 明确继承人的权利范围和限制 | 合同法 |
| 管辖法律 | 根据用户所在地确定适用法律 | 国际私法 |
| 争议解决 | 仲裁优先，适用法院管辖 | 合同法 |

## 10. 法律风险清单

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 用户主张产品内容构成合法遗嘱 | 中 | 高 | 免责声明 + 律师合作服务 |
| 继承人之间的继承权纠纷 | 高 | 中 | 仅执行用户设定，不参与纠纷 |
| 未成年继承人的法律代理问题 | 中 | 中 | 要求监护人作为代理接收人 |
| 用户存储违法内容 | 低 | 高 | 零知识（我们技术上无法审查）+ 用户协议免责 |
| 跨境数据传输违规 | 中 | 高 | 分区存储 + SCCs + 安全评估 |
| 误触发继承（用户仍存活） | 低 | 极高 | 5阶段渐进验证 + 多人确认 + 撤销机制 |
| 政府要求解密数据 | 中 | 高 | 零知识架构（技术上无法解密）+ 透明度报告 |
| 数字遗产法律变化 | 中 | 中 | 法律监控 + 架构灵活性 |
