# Legacy Vault - 产品设计文档

## 1. 产品概述

Legacy Vault 是一个军事级加密的数字保险箱应用。用户可以存储一生中最私密的信息——密码、遗嘱、秘密、私人信件、给后人的话——并设定精确的继承规则，确保在正确的时刻将正确的内容交给正确的人。

**核心价值主张**：在绝对安全的前提下，让数字遗产的传承充满人性温度。

## 3. 核心用户旅程

### 旅程A：首次注册与设置（所有用户）
```
Landing Page → 了解产品价值 → 注册账号
→ 创建主密码（强密码引导 + 安全提示）
→ 生成恢复密钥（Recovery Kit）
→ 引导设置第一个保险箱条目
→ 提示设置紧急联系人（可跳过）
```
**关键体验点**：
- 注册必须在5分钟内完成
- 主密码创建需要引导但不能让用户感到被"教育"
- 恢复密钥的重要性必须强调，但方式不能恐吓用户

### 旅程B：日常使用（密码管理器）
```
打开应用/浏览器扩展 → 生物识别或主密码解锁
→ 浏览/搜索密码 → 一键自动填充
→ 新增密码（手动或自动捕获）
→ 安全审计提醒（弱密码/重复密码/泄露密码）
```
**与1Password/Bitwarden的差异化**：
- 每条密码可独立设置继承人和触发条件
- "安全健康分"中包含"遗产覆盖率"——有多少账号已设置继承方案
- 支持附加"备注遗言"：给继承人的操作说明（如"这个账号里有我们的照片，请保留"）

### 旅程C：存储私密内容
```
打开保险箱 → 选择类型（文字/文件/照片/视频/录音）
→ 编辑/上传内容 → 客户端加密 → 安全存储
→ 设置访问权限（仅自己 / 指定继承人 / 紧急访问可见）
→ 设置触发条件（立即可访问 / 我去世后 / 指定日期 / 指定事件）
→ 添加备注（给继承人的说明）
```

### 旅程D：设置遗产计划
```
进入"遗产规划"模块 → 添加继承人（邮箱/手机号邀请）
→ 继承人接受邀请并完成身份验证
→ 分配内容（拖拽式分配界面）
    - 给妻子：所有密码、银行信息、保险单
    - 给儿子：个人信件、给他18岁的视频
    - 给律师：遗嘱文本、资产清单
→ 设置触发条件
    - 主触发：生存验证超时（30天未响应）
    - 确认：至少2名继承人确认
    - 备用：律师出具死亡证明
→ 预览继承人将看到的内容
→ 确认并激活遗产计划
```

### 旅程E：生存验证协议（详细流程）
```
系统定期发送"存活确认"请求（用户可配置周期：7/14/30/90天）
→ 用户收到通知（推送/邮件/短信 多渠道）
→ 用户确认存活（一键确认 / 生物识别 / 打开App即确认）
→ 重置计时器

如果未响应：
→ 第1阶段（宽限期）：增加通知频率，多渠道提醒（3天）
→ 第2阶段（升级提醒）：联系用户预设的"提醒联系人"（非继承人），请其提醒用户（7天）
→ 第3阶段（预触发）：通知紧急联系人，询问用户状态（14天）
→ 第4阶段（确认期）：要求至少N名确认人确认用户已故/无行为能力（7天等待期）
→ 第5阶段（执行）：启动继承协议，分发加密内容

任何阶段用户重新响应 → 立即取消流程，重置计时器
```
**防误触发设计**：
- 多阶段渐进式确认，总时长不少于31天
- "提醒联系人"机制避免用户仅因忘记确认而触发
- 确认期需要多人协同确认
- 用户可设置"静默期"（如长途旅行时暂停验证）

### 旅程F：紧急访问协议（详细流程）
```
紧急联系人发起"紧急访问请求"
→ 系统通知用户："XXX正在请求紧急访问你的保险箱"
→ 等待期（用户预设：24小时/48小时/72小时/7天）
    - 用户在等待期内拒绝 → 访问被拒，通知请求者
    - 用户无响应 → 进入下一步
→ 要求额外确认人确认（如设置了M-of-N确认）
→ 请求者通过身份验证（生物识别 + 短信验证码）
→ 解锁用户预先授权的"紧急访问"级别内容
    - 基础级：紧急联系方式、医疗信息
    - 标准级：密码、财务信息
    - 完整级：所有内容
→ 通知用户（所有渠道）访问已发生
→ 记录完整审计日志
```

### 旅程G：继承人接收遗产
```
继承人收到通知："Legacy Vault有一份内容等待你接收"
→ 继承人登录/注册Legacy Vault
→ 身份验证（预设的验证方式：邮箱+手机+身份证/护照）
→ 查看被分配的加密内容列表
→ 输入自己持有的密钥分片
→ 系统验证足够的分片已收集（Shamir阈值达成）
→ 内容解密并展示
→ 继承人可选择：下载/保存到自己的保险箱/删除
```

## 4. 用户故事（User Stories）与验收标准

### Epic 1：账号与安全

**US-1.1** 作为新用户，我希望能在5分钟内完成注册和初始设置，以便快速开始使用。
- AC1：注册流程不超过5个步骤
- AC2：支持邮箱+密码或社交登录
- AC3：主密码强度检测实时反馈
- AC4：注册完成后自动生成恢复密钥并强制用户确认已保存

**US-1.2** 作为用户，我希望用生物识别（指纹/面容）快速解锁应用，而不是每次输入主密码。
- AC1：支持Touch ID / Face ID / Android指纹
- AC2：生物识别失败3次自动回退到主密码
- AC3：关键操作（如修改遗产计划）仍需主密码确认

**US-1.3** 作为用户，我希望能生成和安全保存恢复密钥，以防忘记主密码。
- AC1：恢复密钥为24个英文单词的助记词
- AC2：支持导出为加密PDF或打印
- AC3：恢复流程需要助记词 + 邮箱验证

### Epic 2：密码管理

**US-2.1** 作为用户，我希望自动保存和填充网站密码。
- AC1：浏览器扩展检测到登录表单时自动提示保存
- AC2：支持Chrome/Firefox/Safari/Edge扩展
- AC3：自动填充延迟不超过500ms
- AC4：支持TOTP两步验证码管理

**US-2.2** 作为用户，我希望为每条密码设置继承人，让指定的人在我无法操作时获取密码。
- AC1：每条密码支持独立设置1-5个继承人
- AC2：可为密码添加"给继承人的备注"
- AC3：继承人设置变更需要主密码确认

**US-2.3** 作为用户，我希望看到密码的安全评估报告。
- AC1：检测弱密码、重复密码
- AC2：对接HaveIBeenPwned API检测泄露密码（以k-匿名方式保护隐私）
- AC3："遗产覆盖率"指标：已设置继承方案的账号占比

### Epic 3：保险箱存储

**US-3.1** 作为用户，我希望加密存储多种类型的内容。
- AC1：支持文字笔记（Markdown格式）
- AC2：支持文件上传（PDF/DOC/XLS等，单文件不超过100MB）
- AC3：支持照片/视频/录音（主流格式）
- AC4：所有内容在上传前客户端加密

**US-3.2** 作为用户，我希望对内容进行分类和搜索。
- AC1：支持文件夹/标签分类
- AC2：支持全文搜索（在客户端解密后搜索）
- AC3：支持按类型、日期、继承人筛选

### Epic 4：遗产规划

**US-4.1** 作为用户，我希望添加和管理继承人。
- AC1：通过邮箱或手机号邀请继承人
- AC2：继承人需接受邀请并完成身份验证
- AC3：支持为继承人设置"信任等级"（基础/标准/完整）
- AC4：支持移除继承人（需主密码确认，移除后密钥分片重新分配）

**US-4.2** 作为用户，我希望灵活配置触发条件。
- AC1：支持"生存验证超时"触发（可配置周期）
- AC2：支持"指定日期"触发（时间胶囊模式）
- AC3：支持"多人确认"触发（M-of-N模式）
- AC4：支持"律师确认"触发（对接公证服务）
- AC5：支持组合触发条件（AND/OR逻辑）

**US-4.3** 作为用户，我希望预览继承人将看到的内容。
- AC1：模拟继承人视角展示分配的内容列表
- AC2：显示每条内容的解锁条件
- AC3：可直接在预览中调整分配

### Epic 5：时间胶囊

**US-5.1** 作为用户，我希望给未来的自己或他人写定时消息。
- AC1：支持文字/图片/视频/录音格式
- AC2：支持设置精确的发送日期和时间
- AC3：发送前可修改或取消
- AC4：到期后通过邮件/推送通知收件人

### Epic 6：紧急访问

**US-6.1** 作为紧急联系人，我希望在紧急情况下能访问用户的保险箱。
- AC1：发起紧急访问请求后，用户收到多渠道通知
- AC2：等待期内用户可一键拒绝
- AC3：等待期过后需通过身份验证
- AC4：仅能访问用户预授权级别的内容
- AC5：完整的审计日志记录

## 6. 安全架构（概要，详见 docs/SECURITY_ARCHITECTURE.md）

### 加密方案
```
用户主密码
    | (Argon2id 派生, m=47104, t=1, p=1)
    v
主密钥 (Master Key, 256-bit)
    |
    v
  +-------------------+
  | 数据加密密钥       | <-- AES-256-GCM 加密每条数据
  | (每条数据独立DEK)  |     每个DEK由主密钥加密后存储
  +-------------------+
    |
    v
  +-------------------+
  | 继承人密钥分片     | <-- Shamir秘密共享 (t, n) 阈值方案
  | (t-of-n 阈值)     |     需要t个分片重建主密钥
  +-------------------+
```

### 零知识架构
- 服务端只存储加密后的数据（密文 + 加密后的DEK）
- 主密码和主密钥永远不离开用户设备
- 服务端无法解密任何用户数据
- 即使整个数据库泄露，攻击者也无法获得明文

### 生存验证协议
```
定期检查(可配置周期) → 用户响应 → 重置计时器
                      | (未响应)
                      v
                 宽限期(3天, 多渠道提醒)
                      | (仍未响应)
                      v
                 联系提醒联系人(7天)
                      | (仍未响应)
                      v
                 通知紧急联系人确认(14天)
                      | (确认人协同确认)
                      v
                 等待期(7天冷却)
                      | (无异议)
                      v
                 启动继承协议，分发内容
```

## 8. 技术选型

| 层 | 技术 | 原因 |
|----|------|------|
| 前端 | React + TypeScript | 安全审计工具丰富，生态成熟 |
| 移动端 | React Native | 访问安全存储API（Keychain/Keystore） |
| 桌面端 | Tauri | 轻量安全，Rust后端 |
| 加密库 | libsodium (NaCl) | 经过广泛验证的加密库 |
| 密钥派生 | Argon2id (via libsodium) | OWASP推荐，抗GPU/侧信道攻击 |
| 浏览器端加密 | Web Crypto API | 浏览器原生，硬件加速 |
| 本地数据库 | SQLCipher | 加密SQLite，成熟稳定 |
| 云存储 | S3兼容 + 客户端加密 | 零知识云存储 |
| 认证 | WebAuthn / FIDO2 | 无密码认证，抗钓鱼 |
| 后端 | Rust | 内存安全，适合安全关键服务 |
| 通知服务 | SendGrid + Twilio | 多渠道通知（邮件+短信） |
| 秘密共享 | 自研（基于libsodium） | Shamir (t,n) 阈值方案 |
